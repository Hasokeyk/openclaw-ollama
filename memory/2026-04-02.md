## OpenClaw + Ollama Docker Stack Configuration (2026-04-02)

### Architecture
- **Separate Containers**: Ollama (official cuda image) and OpenClaw (Node 22 slim).
- **Networking**: Socat bridges used inside the OpenClaw container to bypass 127.0.0.1 limitations.
    - External access: 0.0.0.0:18789 -> 127.0.0.1:18790 (Gateway)
    - Ollama link: 127.0.0.1:11434 -> ollama:11434
- **Persistence**: pp/ollama for models, pp/openclaw for config/sessions.

### Technical Learnings
- **OpenClaw Port Mapping**: OpenClaw often defaults to 127.0.0.1. Directly editing config.json sometimes fails; socat is a more robust bridge for Docker environments.
- **UI Build**: OpenClaw UI assets must be built during the Docker image creation phase (pnpm run ui:build) to avoid high RAM usage and crashes during container startup.
- **Pairing/Authorization**: Headless authorization for Telegram is best handled by manually injecting user data into /root/.openclaw/devices/paired.json.
    - Required format: {"telegram:USER_ID": {"id":"telegram:USER_ID", "role":"operator", ...}}
- **Model Compatibility**: Qwen 2.5 7B and Qwen 2.5 Coder 7B show high compatibility with OpenClaw's tool-calling schema compared to Llama 3.1 8B.
- **Gemini Integration**: Best handled via the OpenAI-compatible endpoint (https://generativelanguage.googleapis.com/v1beta/openai/) to avoid model naming conflicts in the native Google provider.

### Credentials
- Managed via .env file at the root level.
- Environment variables: TELEGRAM_BOT_TOKEN, TAVILY_API_KEY, GOOGLE_GENERATIVE_AI_API_KEY, OPENCLAW_CHANNELS_TELEGRAM_TOKEN.
